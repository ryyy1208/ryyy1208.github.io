以下是严格按照你提供的代码顺序，对每一个参数的详细解析。这些信息直接基于你提供的脚本代码 `protein_mpnn_run.py`。

### 1. 基础打印与模式控制

- **`--suppress_print`**
    
    - **含义**：是否抑制控制台的打印输出。
        
    - **如何使用**：传参 `0`（默认，打印信息）或 `1`（关闭打印）。
        
    - **何时使用**：在批量处理海量数据或将其作为子模块被其他脚本调用时使用，避免终端被无用信息刷屏。
        
    - **影响**：设置为 1 时，代码运行过程中的进度和提示信息将被隐藏。
        
- **`--ca_only`**
    
    - **含义**：仅解析 CA（阿尔法碳）原子的结构，并使用仅基于 CA 的模型。默认为 False。
        
    - **如何使用**：命令行加上 `--ca_only` 即可触发。
        
    - **何时使用**：当你的输入 PDB 只有 CA 骨架（例如某些粗粒度模拟结果），或者全原子骨架质量极差时。
        
    - **影响**：程序将不再寻找 N、C、O 原子，而是加载专门针对 CA 训练的模型权重。
        

### 2. 模型权重与版本

- **`--path_to_model_weights`**
    
    - **含义**：模型权重文件夹的路径。
        
    - **如何使用**：`--path_to_model_weights /path/to/weights/`。
        
    - **何时使用**：每次运行通常都需要指定（除非权重文件与脚本在默认的同级相对路径下）。
        
    - **影响**：决定了程序加载哪个目录下的预训练模型。
        
- **`--model_name`**
    
    - **含义**：ProteinMPNN 模型的具体名称，默认为 `v_48_020`。
        
    - **如何使用**：可输入 `v_48_002`, `v_48_010`, `v_48_020` 或 `v_48_030`。
        
    - **何时使用**：根据你输入骨架的质量选择。例如 `v_48_010` 代表模型使用 48 条边并在训练时加入了 0.10Å 的噪声。
        
    - **影响**：数字越大（如 030），模型对非天然、不完美骨架的容错率越高，但生成的序列可能偏离严格的物理天然性。
        
- **`--use_soluble_model`**
    
    - **含义**：加载仅在可溶性蛋白质上训练的 ProteinMPNN 权重。默认为 False。
        
    - **如何使用**：加上 `--use_soluble_model` 触发。
        
    - **何时使用**：明确需要设计水溶性蛋白时。
        
    - **影响**：模型会极大地减少在表面生成疏水氨基酸（如亮氨酸、异亮氨酸），避免像跨膜蛋白那样发生聚集。
        

### 3. 全局设置与输出选项

- **`--seed`**
    
    - **含义**：随机种子。
        
    - **如何使用**：传参具体整数，如 `--seed 42`。如果设为 0（默认），程序将自动挑选一个随机种子。
        
    - **何时使用**：需要结果具备严格的可复现性时，设定一个固定的非 0 整数。
        
- **`--save_score`**
    
    - **含义**：将得分（负对数概率，-log_prob）保存到 npy 文件中。
        
    - **如何使用**：设为 1 开启，默认为 0（关闭）。
        
    - **影响**：在输出目录下额外生成包含每个位置得分的矩阵文件。
        
- **`--save_probs`**
    
    - **含义**：保存 MPNN 预测的每个位置的氨基酸概率分布。
        
    - **如何使用**：设为 1 开启，默认为 0。
        
    - **影响**：输出文件将包含 21 维的概率矩阵，常用于后续的序列 Logo 图绘制或二次分析。
        

### 4. 仅评估模式（不生成新序列）

- **`--score_only`**
    
    - **含义**：仅对输入的“骨架-序列”对进行打分，不生成新序列。
        
    - **如何使用**：设为 1 开启，默认为 0。
        
- **`--path_to_fasta`**
    
    - **含义**：在仅打分模式下，提供输入序列的 fasta 文件路径。
        
    - **如何使用**：格式应如 `GGGGGG/PPPPS/WWW`，表示按字母顺序排序并用 `/` 分隔的 A、B、C 链。
        
- **`--conditional_probs_only`**
    
    - **含义**：仅输出条件概率，即给定其余序列和骨架的情况下，某个位置 $s_i$ 的概率。
        
- **`--conditional_probs_only_backbone`**
    
    - **含义**：如果为真，输出给定骨架情况下的条件概率。
        
- **`--unconditional_probs_only`**
    
    - **含义**：在一次前向传播中输出给定骨架情况下的无条件概率。
        
    - _以上三个 `probs` 参数通常在你想深度剖析模型在特定位置对不同氨基酸的偏好时使用。_
        

### 5. 采样参数

- **`--backbone_noise`**
    
    - **含义**：添加到骨架原子上的高斯噪声的标准差。默认 0.00。
        
    - **如何使用**：例如 `--backbone_noise 0.05`。
        
    - **何时使用**：骨架不准确或希望增加采样的多样性时。
        
- **`--num_seq_per_target`**
    
    - **含义**：为每个目标骨架生成的序列总数。默认为 1。
        
    - **如何使用**：通常设置为 8, 16 或更高。
        
- **`--batch_size`**
    
    - **含义**：批处理大小，默认为 1。
        
    - **影响**：对于高端 GPU（如 Titan, Quadro），调高此值可大幅缩短总体生成时间；若显存溢出则需调小。
        
- **`--max_length`**
    
    - **含义**：支持的最大序列长度，默认 200000。防止极长序列导致 OOM。
        
- **`--sampling_temp`**
    
    - **含义**：氨基酸采样的温度系数，默认为 "0.1"。
        
    - **如何使用**：可以传入多个值如 "0.1 0.2 0.25"。
        
    - **影响**：建议值在 0.1 到 0.3 之间。值越大，生成的序列突变越多、多样性越丰富，但折叠稳定性可能下降。
        

### 6. 路径与目标对象

- **`--out_folder`**
    
    - **含义**：输出序列的文件夹路径。
        
- **`--pdb_path`**
    
    - **含义**：需要设计的单个 PDB 文件路径。
        
- **`--pdb_path_chains`**
    
    - **含义**：定义该单一 PDB 中需要被重设计的特定链（例如 "A B"）。
        
- **`--jsonl_path`**
    
    - **含义**：包含已解析为 jsonl 格式的 PDB 数据文件夹。用于大批量任务加载。
        

### 7. 序列约束与偏好调整

- **`--chain_id_jsonl`**
    
    - **含义**：指定哪些链被设计、哪些链固定的字典文件路径。如果不提供，默认设计所有链。
        
- **`--fixed_positions_jsonl`**
    
    - **含义**：指定氨基酸序列中绝对不能改变的固定位置的字典文件路径。用于保护关键催化位点。
        
- **`--omit_AAs`**
    
    - **含义**：在生成序列时全局剔除的氨基酸，默认为 'X'。
        
    - **如何使用**：如传入 'AC' 将完全不生成丙氨酸和半胱氨酸。
        
- **`--bias_AA_jsonl`**
    
    - **含义**：控制全局氨基酸组成偏好的字典文件路径。
        
    - **影响**：例如 `{A: -1.1, F: 0.7}` 会人为降低 A 的生成概率并提高 F 的生成概率。
        
- **`--bias_by_res_jsonl`**
    
    - **含义**：为每一个特定残基位置单独设定氨基酸偏好分数的字典。
        
- **`--omit_AA_jsonl`**
    
    - **含义**：在特定链的特定索引位置上，禁止生成某些特定的氨基酸的字典文件。
        

### 8. PSSM (位置特异性打分矩阵) 控制

_以下参数均用于结合进化信息（PSSM）指导序列生成：_

- **`--pssm_jsonl`**：包含 PSSM 分数的字典路径。
    
- **`--pssm_multi`**：权重控制，值在 [0.0, 1.0] 之间。0 代表不用 PSSM，1 代表完全忽略 MPNN 的深度学习预测。
    
- **`--pssm_threshold`**：设定阈值限制每一位的氨基酸类型。
    
- **`--pssm_log_odds_flag`** / **`--pssm_bias_flag`**：设置为 1 时，激活基于 PSSM 对数几率或偏差调整生成概率的算法逻辑。
    

### 9. 对称性与多聚体控制

- **`--tied_positions_jsonl`**
    
    - **含义**：指定哪些位置必须“绑定”的字典文件路径。
        
    - **何时使用**：在设计同源多聚体（如要求链 A、链 B 序列必须完全同步）时，通过此参数强制不同链对应位置突变一致。

对于 ProteinMPNN，你完全不需要担心必须在黑框框（命令行）里敲代码。虽然它的核心是一个 Python 项目，但社区已经为它开发了多种**友好的 GUI（图形用户界面）**。

你可以根据你的需求选择以下三种方式：

---

### 1. 云端 GUI：最推荐（无需配置，开箱即用）

如果你不想折腾环境配置，以下在线平台提供了完整的图形化操作界面：

- **Hugging Face Spaces (by Simon Duerr):**
    
    这是目前最流行的可视化版本。你可以直接上传 PDB 文件，在网页上通过滑块调整参数（如采样温度 `Temperature`、固定残基等），然后一键生成序列。
    
- **ColabDesign (Google Colab):**
    
    这是由 Sergey Ovchinnikov 等人维护的项目。虽然它基于 Jupyter Notebook，但提供了非常直观的交互表单。你只需点击“播放”按钮，然后在输入框里填入你的 PDB 信息即可。
    
- **Neurosnap / Tamarind Bio:**
    
    这些是第三方的生物计算平台，它们将 ProteinMPNN 封装成了专业的 Web 工具，界面非常精美，适合进行批量处理和管理。
    

### 2. 交互式界面：功能最全

如果你希望在设计序列的同时直接验证其结构，**ColabDesign** 的 Notebook 版本是最佳选择。

- **优点：** 可以在生成序列后，直接在同一个页面调用 AlphaFold2 进行验证（即 Forward Folding 验证）。
    
- **可调参数：** 你可以可视化地选择哪些残基是“固定不变”的，哪些是需要重设计的。
    

### 3. 本地部署（含简单 GUI）

如果你有自己的 GPU 想要本地运行，但又想要个界面，可以这样做：

1. 从 GitHub 克隆 `ProteinMPNN` 仓库。
    
2. 安装 `Gradio` 库（Python 的 GUI 框架）。
    
3. 运行社区提供的 `app.py` 脚本（如 Simon Duerr 仓库里的），这会在你本地浏览器里生成一个和 Hugging Face 一模一样的操作界面。

### ProteinMPNN 操作界面的核心参数速查

无论你使用哪个 GUI，通常会看到以下几个关键选项：

|**参数名称**|**建议设置**|**作用说明**|
|---|---|---|
|**Sampling Temperature**|0.1 ~ 0.2|越低，序列越稳定（保守）；越高，序列越多样化。|
|**Fixed Positions**|按需填写|比如：`A1-10, B20`。指定哪些位置的氨基酸不允许改变。|
|**Homo-oligomer**|勾选/不勾选|如果你的蛋白质是多聚体（如同源二聚体），勾选此项可确保对称性。|
|**Number of Sequences**|8 ~ 32|一次生成的候选序列数量。|

我当然知道。如果你在关注蛋白质设计（Protein Design）或者结构生物学，**ProteinMPNN** 绝对是目前该领域“大明星”级别的存在。

简单来说，它是华盛顿大学 David Baker 实验室（2024年诺贝尔化学奖得主）开发的一种基于深度学习的**蛋白质序列设计**工具。

以下是关于它的核心干货：

---

### 1. 它解决什么问题？

在蛋白质设计中，有一个核心挑战叫**“逆折叠问题” (Inverse Folding)**：

- **传统问题：** 给定序列，预测结构（如 AlphaFold2）。
    
- **ProteinMPNN 的问题：** 我已经有了一个理想的骨架结构（Backbone），什么样的氨基酸序列能稳定地折叠成这个样子？
    

### 2. 为什么它这么出名？

在 ProteinMPNN 出现之前，大家主要用 RosettaDesign。相比之下，ProteinMPNN 有几个碾压级的优势：

- **速度极快：** 秒级生成序列，比传统基于物理计算的方法快上千倍。
    
- **成功率高：** 实验证明，它设计的序列在湿实验（实验室合成）中的折叠成功率显著高于以往工具。
    
- **鲁棒性强：** 它对骨架结构的微小噪声不敏感，这让它非常适合处理由 AI（如 RFdiffusion）生成的“人造骨架”。
    

### 3. 工作原理简述

ProteinMPNN 本质上是一个**消息传递神经网络（Message Passing Neural Network）**。

- **输入：** 蛋白质骨架原子的 3D 坐标（C, Cα, N, O）。
    
- **过程：** 它将蛋白质看作一个图（Graph），每个残基是节点，空间邻近的残基通过边连接。通过在图中传递信息，模型能够学习到残基之间的环境关系。
    
- **输出：** 每个位置上最匹配的氨基酸概率分布。
    

### 4. 黄金搭档：RFdiffusion + ProteinMPNN

现在蛋白质设计界最流行的“工作流”通常是这样的：

1. **RFdiffusion：** 凭空生成一个炫酷的蛋白质 3D 骨架。
    
2. **ProteinMPNN：** 给这个骨架“填肉”，设计出能撑起这个结构的氨基酸序列。
    
3. **AlphaFold2 / ProteinMPNN-Validation：** 验证序列是否真的能折叠回预期的结构。
    

---

**一句话总结：**

如果说蛋白质设计是一场建筑工程，**RFdiffusion 是绘图员**，负责画蓝图；而 **ProteinMPNN 就是材料员**，负责挑选最合适的钢筋和混凝土（氨基酸）来把房子盖起来。
